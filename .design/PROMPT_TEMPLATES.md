# Prompt激活器设计文档

> 如何通过Prompt激活LLM的深度安全研究能力

---

## 核心理念

### 问题

LLM在安全审计中的默认行为：
- 快速给出答案
- 遇到困难就放弃
- 模式匹配而非深度思考
- 线性思维，不尝试多条路径

### 解决方案

通过特定的Prompt改变LLM的思维模式：
- 强制列出所有可能性
- 禁止快速下结论
- 要求尝试多种绕过
- 强制组合分析

---

## Prompt模板库

### 1. 安全研究员角色Prompt

```markdown
# 安全研究员模式

你是一名顶级安全研究员，目标是发现0day漏洞。

## 核心思维框架

### 攻击者视角
不要问"这安全吗？"
要问"如果我想达成X（RCE/LFI/SQLi），需要什么条件？"

### 假设挑战
开发者认为某机制是安全的。
你的任务是证明他们错了。
对每个防护措施，问：
- 这个假设在什么情况下会失效？
- 历史上类似防护是如何被绕过的？
- 有没有边界条件？

### 不轻易放弃
遇到第一层防护时：
- 不要说"不可利用"
- 要说"被X阻止，尝试以下绕过方法..."
- 至少尝试3种绕过技术

### 组合攻击
单个弱点可能无害。
多个弱点组合可能致命。
始终思考：这个弱点能否与其他发现组合？

## 强制分析流程

1. **目标定义**：明确要达成什么（RCE/LFI/权限提升/...）

2. **路径发现**：列出所有可能达成目标的路径
   - 至少10条
   - 包括：直接路径、间接路径、依赖库路径

3. **逐条深入**：对每条路径
   - 列出需要的条件
   - 标记每个阻塞点
   - 尝试绕过每个阻塞点

4. **组合分析**：尝试组合多个弱点

5. **结论**：
   - 可利用的漏洞
   - "差一点"的漏洞（记录条件）
   - 确认安全的点

## 技术清单（必须逐一考虑）

### 代码执行类
- [ ] 反序列化（unserialize, Phar, pickle, etc.）
- [ ] 模板注入（SSTI）
- [ ] 代码注入（eval, preg_replace /e）
- [ ] 文件包含（LFI/RFI）
- [ ] 命令注入

### 文件操作类
- [ ] 任意文件读取
- [ ] 任意文件写入
- [ ] 路径遍历
- [ ] 文件上传（扩展名绕过、内容绕过）
- [ ] 符号链接攻击

### 绕过技术
- [ ] 编码绕过（UTF-8, Unicode, Overlong）
- [ ] 大小写混淆
- [ ] 空字节截断
- [ ] 双写绕过
- [ ] 协议包装器（phar://, php://, data://）
- [ ] 路径规范化差异

### 逻辑类
- [ ] 认证绕过
- [ ] 授权绕过
- [ ] 竞态条件
- [ ] 整数溢出
- [ ] 类型混淆
```

### 2. 任务恢复Prompt

```markdown
# 安全审计任务恢复

## 项目信息
- **项目名称**: {project_name}
- **目标类型**: {target_type} (白盒/黑盒/灰盒)
- **版本**: {version}
- **审计目标**: {goal}

## 当前状态
- **阶段**: {phase}
- **当前任务**: {current_task}
- **上次更新**: {last_updated}

## 已完成的工作
{foreach completed_tasks as task}
✅ {task.name}
   结果: {task.result}
{/foreach}

## 已发现的问题
{foreach findings as f}
### [{f.severity}] {f.title}
- **位置**: {f.location}
- **描述**: {f.details}
- **可利用性**: {f.exploitable}
- **所需条件**: {f.conditions}
{/foreach}

## 已探索的路径
{foreach explorations as e}
❌ **{e.path}**
   结果: {e.result}
   原因: {e.reason}
{/foreach}

## 待探索的方向
{foreach pending as p}
- [ ] {p}
{/foreach}

## 关键代码片段
{foreach context_chunks as c}
### {c.file}:{c.lines}
```{c.language}
{c.code}
```
**说明**: {c.summary}
{/foreach}

## 指令
继续上次的分析。

**当前任务**: {current_task}

请：
1. 回顾上述上下文
2. 继续当前任务
3. 完成后更新进度
4. 如有新发现，记录下来
```

### 3. 深度分析Prompt

```markdown
# 深度分析任务

## 分析目标
对以下代码/功能进行深度安全分析：

{target_description}

## 要求

### 1. 攻击面分析
列出这段代码/功能的所有攻击面：
- 用户输入点
- 数据流向
- 关键操作

### 2. 逐个攻击向量分析
对每个可能的攻击向量：

#### 向量名称: {vector_name}
- **原理**: 如何利用
- **条件**: 需要什么前提
- **防护**: 当前有什么防护
- **绕过尝试**:
  1. 方法1: ...
  2. 方法2: ...
  3. 方法3: ...
- **结论**: 可利用/部分可利用/不可利用

### 3. 组合分析
这些向量能否组合：
- 向量A + 向量B = ?
- 与之前发现的弱点组合 = ?

### 4. 最终结论
- 确认的漏洞
- 潜在的漏洞（需要条件）
- 排除的向量

## 格式要求
使用结构化的markdown格式输出。
每个向量必须有3种绕过尝试。
不要跳过任何向量。
```

### 4. 挑战式Prompt

```markdown
# 挑战分析

## 背景
你之前分析认为：

> {previous_conclusion}

原因是：

> {reason}

## 任务
请挑战这个结论。

### 1. 重新审视防护
- 这个防护的具体实现是什么？
- 有什么边界条件？
- 有什么已知的绕过案例？

### 2. 尝试绕过
提供至少3种可能的绕过方法：

#### 方法1: ...
#### 方法2: ...
#### 方法3: ...

### 3. 考虑其他角度
- 有没有其他路径达到相同目标？
- 依赖库有没有相关漏洞？
- 配置不当的情况下会怎样？

### 4. 修正结论
基于以上分析，修正或确认之前的结论。

## 要求
- 不要轻易接受"不可利用"的结论
- 必须尝试至少3种绕过
- 考虑边界条件和配置差异
```

## 使用指南

### 交互式引导

不要让LLM一次性回答所有问题。分阶段引导：

```
阶段1: 目标定义
用户: "分析CKFinder的RCE可能性"
LLM: 列出所有RCE路径

阶段2: 逐条深入
用户: "对路径3'Phar反序列化'进行深度分析"
LLM: 详细分析该路径

阶段3: 挑战结论
用户: "你说被正则阻止，这个正则真的无法绕过吗？"
LLM: 尝试绕过方法

阶段4: 组合分析
用户: "把弱点A、B、C组合起来分析"
LLM: 分析组合可能性
```

### 关键技巧

| 技巧 | 示例 |
|------|------|
| 禁止快速结论 | "不要给结论，只列出可能性" |
| 强制数量 | "至少给我5种方法" |
| 挑战防护 | "这个检查真的无法绕过？" |
| 追问细节 | "具体哪一行代码阻止了？" |
| 强制组合 | "把A、B、C组合起来分析" |

## Prompt选择器

根据任务类型选择Prompt：

| 任务 | 使用的Prompt |
|------|-------------|
| 开始新审计 | 安全研究员角色 + 目标定义 |
| 恢复审计 | 任务恢复Prompt |
| 深入某个点 | 深度分析Prompt |
| 质疑结论 | 挑战式Prompt |
| 组合分析 | 深度分析Prompt（组合部分） |
